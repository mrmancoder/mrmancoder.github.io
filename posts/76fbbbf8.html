讲讲事务的基本知识

Mr.Man2025-08-102025-08-10

注：本篇文章为个人学习所用，可能存在不严谨或出错的地方，还请谅解。

1. 什么是事务?

1.1. 什么是事务?

​事务,是一个完整的执行体,一个事务内所有操作被看作一个整体,具有相同的生命周期,不可被拆分.

现在以银行转账的案例分析事务的特性,我要给小明转100元,可以将这件事拆成:我的账户扣款100元;小明的账户增加100元.

先来讲事务的第一个特性,即原子性,原子性要求将事务看成最小的一个原子操作,不可再拆分,要么全部成功,要么全部失败,不会在执行过程中结束掉.以转账例子分析,转账过程要么成功,要么失败,不会出现我的账户扣100元,但小明账户没有增加100元的情况.原子性的核心是不能让事务在中间断掉.

事务的第二个特性是一致性,讲的是事务执行前后,数据是一致的,以转账为例,转账前我和小明的账户一共有1000元,转账后,我们的账户还是有1000元,不会出现我的账户扣钱,小明账户没有进钱的情况.

事务的第三个特性是隔离性,隔离性可以让不同的事务同时对数据进行操作和修改,核心是每个事务都有一个独立的数据空间,保证不同事务在执行时互不干扰,即我在买一件商品时,不影响其他人也买这件商品.

事务在并行执行时,会出现脏读等问题,隔离性包括4种隔离方案,可以有效的解决脏读等问题,会在下文讲解.

事务的最后一个特性是持久性,指的是事务提交后,对数据的操作是永久有效的.

1.2. 事务并行存在的问题?

以事务A和事务B并行执行举例,分别讲下脏读、不可重复读、幻读三个场景.

  • 脏读: 事务A处理了数据,但还未提交事务,事务B读到了事务A未提交时修改的数据.

假设两个事务同时执行,处理账户余额,事务A先将余额从100改为200,暂未提交事务,此时事务B读取余额,读到200,这时候如果事务A发生回滚,余额回滚为100,造成事务B读到的和实际余额不同.

  • 不可重复读: 一个事务多次读取同一个数据,但读取到的数据不一致,为不可重复读现象.

假设事务A和事务B并发执行,事务A先读取数据为100,此时事务B修改数据为200,然后事务A再次读取数据为200,前后数据不一致.

  • 幻读: 一个事务多次查询符合条件的记录数量,前后读取到的结果不一致,为幻读现象.

假设事务A和事务B并发执行,事务A读取余额>100的记录数量,之后事务B新增了一条余额大于100的记录,事务A再读取记录数量时,发现数量不一致.

2. 事务的隔离级别?

为了解决事务并行带来的问题,数据库提出了4种事务隔离级别,能够灵活的解决并行带来的问题.

  1. 读未提交: 指一个事务还未提交,其修改的数据可以被其他事务读到;
  2. 读已提交: 指一个事务只有提交之后,其修改的数据才可以被其他事务读到;
  3. 可重复读: 指一个事务在执行过程中,读到数据和事务启动时的数据是一致的;
  4. 串行化: 事务执行时会给记录上锁,只有等到前一个事务执行结束,才能执行下一个,严格保证顺序.

不同的隔离级别可解决不同的并发问题:

  • 读已提交: 可以解决脏读问题;
  • 可重复读: 可解决脏读和不可重复读问题(和事务启动时看到的数据一致);
  • 串行化: 可解决脏读、不可重复读、幻读问题.

总结: 从读未提交到串行化,能解决的并发问题越来越多,但是随着隔离级别的提高,对性能的影响越来越大,MySQL默认的隔离级别为可重复读.

3. Read View是什么?

读已提交和可重复读是基于MVCC实现的,而MVCC又是基于Read View完成的,所以先来讲下Read View是怎么工作的.

可以把Read View理解为一个数据快照,即数据库中的视图,以可重复读举例,在事务启动后,会创建一个Read View快照,在事务执行过程中,看到的数据都来自这个Read View,即使其他事务对数据做了修改,也不影响该Read View,使得事务看到的数据和事务启动时看到的数据是一致的.

Read View在创建时,包含了四个字段:

  1. m_ids ：指的是创建Read View时,当前数据库中活跃事务的列表,即已启动,但未提交的事务集合;
  2. min_trx_id ：创建Read View时,活跃事务列表中最小的事务id;
  3. max_trx_id ：创建Read View时,应该给下一个待开启的事务分配的事务id,即m_ids列表中最大的事务id+1;
  4. creator_trx_id ：创建该Read View的事务的id.

在MySQL的聚簇索引中,每个记录都会维护两个和事务相关的隐藏列,如图所示

  • 一个是事务id,表示最新修改当前记录的事务id;
  • 一个是回滚指针,指向的是上一个版本的内容,用于回滚时,可以找到上一个状态的内容.

每次创建Read View,都会维护一个事务状态的范围,如下图所示,结合Read View几个字段的概念理解这张图,为了更好的说明事务之间的可见性,举例:m_ids:[11,20],min_trx_id=11,max_trx_id=21.

如果一个事务A访问某个记录时,根据记录中的事务id字段,判断当前版本记录的可见性:

  1. 如果事务id<min_trx_id,说明最新修改该记录的事务已经成功提交了,那么当前版本的记录对事务A是可见的;
  2. 如果事务id>=max_trx_id,说明该版本的记录是创建Read View之后开启的事务生成的,该版本的记录对事务A是不可见的;
  3. 如果事务id的范围为: min_trx_id=<id<max_trx_id,需要具体判断事务id是否在活跃事务内:
    • 如果事务id在m_ids范围内,说明事务还没有提交,所以该版本的记录对事务A是不可见的;
    • 如果事务id不在m_ids范围内,说明事务已经提交,所以该版本的记录对事务A是可见的.

4. 可重复读是如何实现的?

隔离级别为[可重复读]的事务,会在事务开始时创建Read View,在事务的整个执行过程,都使用这个Read View.

假设现在有事务A和事务B,事务A先创建,事务id为10,事务B后创建,事务id为11.

现在事务A和事务B处理一条数据,此时指向上一版本的指针为null.

现在假设场景:

  1. 事务B访问该记录,得到money=100;
  2. 事务A修改money为200,但未提交;
  3. 事务B第二次读取该记录,得到money=100;
  4. 事务A提交事务;
  5. 事务B第三次读取该记录,得到money=100.

现在结合Read View讲下这个过程:

  1. 事务B访问该记录,得到事务id=9,小于min_trx_id,说明该记录的操作在事务B创建之前就已经提交了,所以该版本的记录是可见的,得到money=100;
  2. 事务A修改记录为200,此时更新后的版本链如下面的图片;
  3. 事务B第二次读取时,仍按照事务开始时的Read View处理,此时读到事务id=10,发现在min_trx_id和max_trx_id之间,还需要判断id=10仍在m_ids范围内,发现在m_ids范围内,说明id=10的事务还没有提交,该版本的修改是不可见的,此时会沿着版本链依次寻找,直到找到了事务id=9的记录,该事务在创建Read View之前就提交了,所以对事务B是可见的,读到了money=100.
  4. 事务A提交事务.
  5. 事务B第三次读取时,仍按照事务开始时创建的Read View去判断,即使事务A已提交,仍会遍历版本链,查到id=9的事务,读到的仍是money=100.

5. 读已提交是如何实现的?

​读已提交是每次在读取数据时,都会创建Read View快照.

还是以上面的场景为例,在事务A提交了之后,事务B第三次读取记录时,会创建一个新的Read View,因为事务A已经提交了,此时m_ids=[11].

​该记录的最新事务id=10,小于min_trx_id,该版本的记录对事务B是可见的,所以此时读取的money=200,这正是读已提交和可重复读的区别.
